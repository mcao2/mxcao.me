Mesos: a platform for fine-grained resource sharing in the datacenter
  • Mesos is a platform for sharing commodity cluster b/w multiple diverse cluster computing frameworks e.g. Hadoop, MPI
    • Benefits
      • improved cluster utilization
      • avoids per-framework data replication by fine-grained resource sharing and enable data locality
        • take turns reading data stored on each machine
      • supports distributed two-level scheduling mechanism called "resource offers" that enable sophisticated schedulers
    • Mesos decideds how many resources to offer to each framework, and leave it to the frameworks to decide which resources to accept and which computations to run on them
    • Experimenttal results show that Mesos can achieve near-optimal data locality when sharing cluster among diverse frameworks and it is scalable, and resilient to failures
  • Intro to the background
    • Cluster computing frameworks are developed to simplify the programming of clusters
      • e.g. MapReduce, Dryad, Pregel...
    • New frameworks continue to emerge and no framework is a panacea that is optimal for every application
      • => org want to run multiple frameworks in the same cluster and pick the best that fit each application
    • Multiplexing a cluster b/w frameworks enable the chance of improved cluster utilization and data sharing of large datasets
      • Two common solutions for sharing a cluster:
        • statically partition the cluster and run one framework per partition
        • allocate a set of VMs to each framework
      • Problems with such solutions
        • hard to achieve high utilization and efficient data sharing due to the mismatch b/w the allocation granularities of these solutions and of existing frameworks
        • e.g. Hadoop and Dryad employ a fine-grained resource sharing model where nodes are subdivided into slots. Jobs are composed of short tasks that are matched to slots
        • no simple way to perform fine-grained sharing across frameworks
    • The authors introduce Mesos, a thin resource sharing layer that enables fine-grained sharing across diverse cluster computing frameworks
      • by giving frameworks a common interface for accessing cluster resources
    • Design question
      • how to build a scalable and efficient system that supports a wide range of both current and future frameworks
      • Challenges
        • each framework has different scheduling needs based on its programming model, communication pattern, task dependencies, and data placement
        • the scheduling system must scale to clusters of tens of thousands of nodes running hundreds of jobs with milions of tasks
        • all apps in the cluster depend on Mesos and therefore the system must be fault tolerant and highly available
    • Design choices
      • implement a centralized scheduler that takes as input framework requirements, resource availability, and organizational policies and computes a global schedule for all tasks
        • pros
          • can optimize scheduling across frameworks
        • cons
          • complexity! the central scheduler would need to provide sufficiently expressive API to capture all frameworks' requirements and solve an online optimization problem for millions of tasks. Even if such a scheduler were feasible, this complexity would have a negative impact on its scalability and resilience
          • we may not have a full specification of frameworks as new frameworks constantly being developed
          • many existing frameworks already implemented their own specialized, sophisticated scheduler and moving this functionality to a global scheduler would require expensive refactoring
      • delegate control over scheduling to the frameworks (this is the one Mesos used, decentralized scheduling model)
        • by defining a new abstraction called "resource offer" that encapsulates a bundle of resources that a framework can allocate on a cluster node to run tasks
        • key idea:
          • Mesos decides how many resources to offer each framework based on organizational policies e.g. fair sharing
          • frameworks decide which resource to accept and which tasks to run on them
        • pros
          • simple and efficient to implement, allowing Mesos to be highly scalable and robust to failures
          • performs surprisingly well in practice, allowing frameworks to meet goals such as data locality nearly perfectly
          • even orgs that only use one framework can use Mesos to run multiple instances/versions of that framework in the same cluster, as a way to isolate production and experimental workloads
          • easy to develop and immediately experiment with new frameworks
          • frees developers to build and runspecialized frameworks targeted at particular problem domains rather than one-size-fits-all abstractions
          • frameworks can therefore evolve faster and provide better support for each problem domain
        • cons
          • may not always lead to globally optimal scheduling
    • Mesos in general
      • 10000 lines of C++
      • use ZooKeeper for fault tolerance
      • ported 3 cluster computing systems to run over Mesos (Hadoop, MPI, Torque batch scheduler)
      • built a new framework on top of Mesos called Spark to validate that specialized frameworks provide value over general ones
        • results show that Spark can outperform Hadoop by 10x in iterative machine learning framework
  • Mesos architecture
    • design philosophy
      • should be a scalable and resilient core for enabling various frameworks to efficiently share clusters
      • should be a minimal interface that enables the above requirements
        • push control to the frameowrks for task scheduling and execution
          • benefits
            • allow frameworks to implement diverse approaches to various problems in the cluster e.g. achieve data locality, deal with faults
            • independently evolve these solutions
            • keep Mesos simple and minimize the rate of changes required for the system and therefore makes it easier to keep Mesos scalable and robust
      • Mesos provides low-level interfaces and we expect high-level libraries to implement common functionality (e.g. fault tolerance) to be built on top of Mesos
        • allow Mesos keep small and flexible, and lets the libraries evolve independently
    • Architecture diagram
      • main components
        • a master process that manages slave daemons running on each cluster node
        • frameworks that run tasks on these slaves
      • Mesos master
        • implements fine-grained sharing across frameworks using resource offers
          • each resource offer is a list of free resources on multiple slaves
          • master determines how many resources to offer to each framework based on org policy
          • org policies are defined via a pluggable allocation module
      • Frameworks
        • scheduler
          • register with the master process to be offered resources
          • select which of the offered resources to use
          • when it accepts offered resources, it passes Mesos a description of the tasks it wants to launch on them
        • executor
          • launched on slave nodes to run the framework's tasks
        • example scheduling process available in figure 3
        • to be a "thin interface" and enable independent framework evolvement, Mesos does not require frameworks to specify their resource requirements or constraints, instead, frameworks can "reject" offers that not meet their requirements/constraints
          • reject unwanted offers and wait for ones that do meet requirements
          • such rejection mechanism enable frameworks to support arbitrary complex resource constraints while still keep Mesos simple and scalable
          • Chllenge:
            • Efficiency
              • a framework may have to wait a long time before it receives a satisfying offer
              • Mesos may have to send an offer to many frameworks before one of them accepts it
              • Solution:
                • frameworks can set "filters", which are boolean predicates specifying that a framework will always reject certain resources
                • e.g. a framework can create a whitelist of nodes it can run on like GPU only
              • Notes
                • filters are just a performance optimization for the resource offer model and the framework still have the ultimate contonrol to reject/accept any resources that cannot experess filter for and to choose which tasks to run on each node
                • when the workload consists of fine-grained tasks, the resource offer model performs surprisingly well even in the absence of filters.
                  • delay scheduling policy (frameworks wait for a limited amount of time to acquire nodes storing their data) yields nearly optimal data locality with a wait time of 1-5 seconds
      • Resource allocation
        • allocation decision is made by pluggable allocation module defined by organizations
        • currently implemented allocation modules
          • fair sharing based on a generalization of max-min fairness for multiple resources
          • strict priorities
        • normally most tasks are short, and Mesos take advantage of this fact and only reallocates resources when tasks finish. this usually happen frequent enough so that new frameworks acquire their share quickly
          • e.g. if a framework's share is 10% of the cluster, it needs to wait approximately 10% of the mean task length to receive its share
          • BUT...
            • If a cluster becomes filled by long tasks, the allocation module can also revoke (kill) tasks
            • => in such cases, Mesos gives this framework a grace period to clean it up
            • revocation decisions are also made by the allocation module
            • Challenge:
              • though killing a task has a low impact on many frameworks, it is harmful to frameworks with interdependent tasks (e.g. MPI)
                • solution: define a "guaranteed allocation" for each framework, which is a quantity of resources that the framework may hold without losing tasks
                • frameworks read this quantity through an API call and it's the responsibility of the allocation module to ensure the guaranteed allocation all be met concurrently
              • to decide when to trigger revocation, Mesos must know which of the connected frameworks would use more resources if they were offered them (i.e. "interests" in resource offers)
                • frameworks can indicate such interest in offers through an API call
      • Resource isolation
        • provide performance isolation b/w framework executors by leveraging existing OS isolation mechanisms
          • but these mechansims are platform-dependent
          • Solution: pluggable isolation modules are used to support multiple isolation mechanisms
        • OS container technology (Linux Containers and Solaris Projects)
          • enable limits on the CPU, memory, network bandwidth, and IO usage of a process tree
      • Optimizations/techniques that make resource offers scalable and robust
        • task scheduling is a distributed process.. it needs to be efficient, and robust to failures
        • 3 mechanisms in Mesos
          • Support short-circuit the rejection process to avoid communication using "filters"
            • e.g. filters:
              • only offer nodes from list L
              • only offer nodes with at least R resources free
            • filters are evaluated in the master process
            • any resource that does not pass a framework's filter is treated exactly like a rejected resource
          • Mesos counts resources offered to a framework towards its allocation of the cluster since a framework may take time to respond to an offer
            • this is a strong incentive for frameworks to respond to offers quickly and to filter resources that they cannot use
          • Mesos can rescind an offer and re-offer the resources to other frameworks if a framework does not respond to an offer for a sufficiently long period of time
      • Fault tolerance of Mesos
        • all the frameworks depend on the master process and therefore it's critical to make the master fualt-tolerant
        • master is designed to be "soft state" so that a new master can completely reconstruct its internal state from information held by the slaves and the framework scheduler
          • the master's only state is the list of active slaves, active frameworks, and running tasks
          • such info is sufficient to compute how many resources each framework is using and run the allocation policy
        • multiple masters are run in a hot-standby ocnfiguration using ZooKeepers for leader election
        • when the active master fails, the slaves and schedulers connect to the next elected master and repopulate its state
        • Mesos reports node failures and executor crashes to frameworks' schedulers and frameworks can then react to these failures using their own policies
        • Mesos allows frameworks to register multiple schedulers to deal with scheduler failures. If a scheduler failed, then another one is notified by the Mesos master to take over
          • frameworks must use their own mechanisms to share state b/w their schedulers
    • Mesos behavior study
      • conclusion:
        • Mesos performs very well when
          • frameworks can scale up and down elstically,
          • tasks durations are homogeneous
            • Mesos can handle the case where task durations are heterogeneous without impacting the performance of frameworks with short tasks
          • frameworks prefer all nodes equally
            • when different frameworks prefer different nodes (e.g. GPU jobs prefer nodes that have GPUs, MPI jobs prefer nodes within the same rack), Mesos can emulate a centralized scheduler that performs fair sharing across frameworks
        • Frameworks are incentivized to improve their performance under Mesos, and these incentives also improve overall cluster utilization
        • Mesos's distributed scheduling model still has some limitations
      • Metrics
        • framework ramp-up time:
          • time a framework takes to achieve its allocation
        • job completion time
          • assume one job perframework, JCT is the time a job takes to complete
        • system utilization
          • total cluster utilization
      • Elastic v.s. Rigid frameworks
        • elastic framework
          • an elastic framework like Hadoop or Dryad can scale its resources up and down (i.e. it can start using the nodes as soon as it acquires them and release the nodes as soon as its task finish
        • rigid framework
          • a rigid framework like MPI can start running its job only after it has acquired a fixed quantity of resources, and cannot scale up dynamically to take advantage of new resources or scale down without a large impact on performance
      • Homogeneous v.s. Heterogeneous task durations
      • Mandatory v.s. Preferred resources
        • mandatory
          • a resource is mandatory if a framework must acquire it in order to run e.g. GPU is mandatory if a framework cannot run without access to it
        • preferred
          • a resource is preferred if a framework performs better using it but can also run using another quivalent resource
          • e.g. a framework may prefer to run on a node that locally stores its data, but may also be able to read the data remotely if it must
      • Assumptions
        • assume that the amount of mandatory resources requested by a framework never exceeds its guaranteed share, this ensures that frameworks will not deadlock waiting for the mandatory resources to become free
        • assume that all tasks have the same resource demands and run on identical slices of machines called "slots", and that each framework runs a single job
  • Limitations of Mesos's distributed scheduling model
    • though the model works well in a range of workloads relevant to current cluster env, it can perform worse than a centralized scheduler in some cases...
      • fragmentation
        • when tasks have heterogeneous resource demands, a distributed collection of frameworks may not be able to optimize bin packing as well as a centralized scheduler
        • the wasted space is bounded by the ratio b/w largest task size and the node size
          • => clusters running larger nodes and smaller tasks within those nodes will achieve high utilization even with distributed scheduling
      • starvation
        • when a cluster is filled by small tasks, a framework with large resource demands may starve because whenever a small task finish, the framework cannot accept the resources freed by it, but other frameworks can...
        • solution: allocation modules can support a "minimum offer size" on each slave, and abstain from offering resources on the slave until this amount is free
      • interdependent framework constraints
        • e.g. esoteric interdependencies b/w frameworks
          • e.g. certain tasks from two frameworks cannot be colocated
          • in such cases only a single global allocation of the cluster performs well
      • framework complexity
        • using resource offers may make framework scheduling more complex...
        • but the authors' point here is that this difficulty is not onerous
          • 1. the frameworks need to know their preferences no matter we use Mesos or a centralized scheduler, they differ in the way we use such preferences
            • in a centralized scheduler, the framework needs to express them to the scheduler
            • in Mesos, the framework uses its own preferences to decide which offer to accept
          • 2. many scheduling policies for existing frameworks are online algorithms, because they cannot predict task times and must be able to handle failures and stragglers. these policies are easy to implement over resource offers
