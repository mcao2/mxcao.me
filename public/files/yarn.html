Apache Hadoop Yarn
  • Limitations of the initial design of Apache Hadoop
    • tightly focused on running massive, MapReduce jobs to process a web crawl
      • tight coupling of a specific programming model with the resource management infrastructure forces developers to abuse the MapReduce programming model
      • centralized handling of jobs' control flow results in endless scalability concerns for the scheduler
  • This paper introduces YARN, the next generation of Hadoop's compute platform
    • decouples the programming model from the resource management infra
    • delegates many scheduling functions like task fault tolerance to per-application components
    • experimental results demonstrates the improvements on efficiency, flexibility etc.
  • Introduction to background
    • motivation
      • the sucess of Apache Hadoop also becomes its biggest curse
        • developers extend the MapReduce model beyond the capabilities of the cluster management substrate, abuse the usage of MR on e.g. forking webservers, gang-scheduled computation of iterative workloads. Dev often resorted to clever workarounds to sidestep the limitations of MR APIs to leverage the physical resources...
      • the limitations and misuses are now better understood by both the academic and open-source communities and motivated an entire class of papers that use Hadoop as a baseline for unrelated environments...
    • YARN is a community effort to move Hadoop past its original incarnation
      • delegates many scheduling related functions to per-job components
        • by separating resource management functions from the programming model
        • In such new context, MR is just one of the applications running on top of YARN
        • enable great flexibility to choose different programming frameworks that run on top of YARN
          • e.g. Dryad, Giraph, Spark, Storm...
        • programming frameworks running on YARN coordinate intro-application communication, execution flow, and dynamic optimizations as they see fit, unlocking dramatic performance improvements
  • Historical account of how YARN's requirements emerged from practical needs
    • Scalability
    • Multi-tenancy
    • Serviceability: decoupling of upgrade dependencies
    • Locality awareness
    • High cluster utilization
    • Reliability/Availability
    • Secure and auditable operation
    • Support for programming model diversity
    • Flexible resource model
    • Backward compability
  • YARN's architecture
    • YARN lifts some functions into a platform layer responsible for resource management, leaving coordination of logical execution plans to a host of framework implementation
    • per-cluster ResourceManager (RM)
      • runs as a daemon on a dedicated machine and acts as the central authority arbitrating resources among competing apps in the cluster (tenants). It handles live resource scheduling and helps central components in YARN scale beyond the Hadoop 1.0 JobTracker
      • expose two public interfaces
        • 1. for client submitting applications
        • 2. for dynamic access negotiation to resources with the AM
          • AM codify their need in terms of one or more ResourceRequests, each request tracks
            • a. # containers
            • b. resources per container
            • c. locality preferences [e.g. node-level, rack-level, global-level preferences]
              • strict or negotiable
            • d. priority of requests within the application
          • such design allows
            • more uniform req to be represented compactly as aggregates
            • impose size limits on ResourceRequests and thus leveraging the roll-up nature of ResourceRequests as a lossy compression of app's preferences (=> efficient communication/storage)
            • guide the scheduler towards good alternatives in case perfectly locality cannot be achieved
      • expose one internal interface
        • with the NMs for cluster monitoring, resource access management.
      • global view assembled from the NM state that included in the heartbeat messages
        • such central, global view helps
          • (tightly) enforce rich, familiar properties such as fairness, capacity, locality across tenants
        • but requires the scheduler to obtain an accurate understanding of app's resource requirements
        • the communication msgs and scheduler state must be compact and efficient for the RM to scale against app demand and cluster size
        • YARN captures the resource requests in a way that strike a balance b/w accuracy and compactness
        • the scheduler only needs to handle the overall resource profile for each app, ignoring local optimizations and internal app flow
        • it treats the cluster resources as a discretized continuum
      • responsibilities:
        • tracks resource usage and node liveness
          • the RM interacts with a special system daemon running on each node called the NodeManager (NM) to enforce/track allocation assignments
          • NM and RM communicate via heartbeat messages for scalability
        • enfoce allocation invariants
          • depends on the app's demand, scheduling priorities, resource availability, the RM dynamically allocate leases in the form of "containers" to applications to run on particular nodes
          • a "container" is a logical bundle of resources (e.g. <2GB RAM, 1CPU>) bound to a particular node
        • arbitrate contention among tenants
        • forwards the exit status of finished containers as reported by the NMs to the responsible AMs
        • notify AMs when a new NM joins the cluster so that they can start requesting resources on the new nodes
        • [new extension] revoke resources from an application when cluster resources are scarce to maintain scheduling invariants
          • AMs have some flexibility to choose which to stop and how e.g. checkpointing the state of a task or migrate it to other running containers, to allow apps to preserve work
          • the RM can focibly terminate containers to grab the resoruces back if the app does not collaborate and our patience (grace period) run out
      • RM is not responsible for:
        • coordiante app execution or task fault-tolerance
        • provide status or metrics for running apps
        • serve framework specific reports of completed jobs
    • ApplicationMaster (AM)
      • AM is a process that run in the cluster just like any other container!
      • it periodically heartbeats to the RM to affirm its liveness and update the record of its demand
      • it also encodes its preferences and constraints in a heartbeat msg to the RM
      • it receives a container lease on bundles of resources bound to a particular node in the cluster
      • The allocations to an application are late binding:
        • Late binding: the process spawned is not bound to the request, but to the lease
        • the conditions that caused the AM to issue the request may not hold when it receives its resources, but the semantics of the container are fungible and framework-specific
        • the AM will also update its resources asks to the RM as the containers it receives affect both its present and future requirements
          • e.g. after a MR AM receives a container from RM, it matches against the set of pending map tasks, select a task with input data close to the container. the AM will then update its preferences for hosts that have replicas of that map task's input data. Also if task fails, AM should update the weights back to compensate.
      • responsibilities
        • coordinates the logical plan of a single job by requesting resources from the RM
        • generate a physical plan from the resources it receives
        • coordinate the execution of that plan around faults
        • determines the semantics of the success or failure of the container exit status reported by NMs through the RM
      • The AM is the head of a job and manages all lifecycle aspects including
        • dynamically increasing/decreasing resource consumption,
        • managing the flow of execution (running reducers against the output of maps)
        • handling faults and computation skew
        • perform other local optimizations
      • AM can run arbitrary user code, and can be written in any programming language since all communication with the RM and NM is encoded using extensible communication protocols
      • such delegation enable scalability, flexibility, improved upgrading/testing (since multiple versions of the same framework can coexist)
    • NodeManager (NM)
      • NM is the "worker" daemon in YARN
      • responsibilities
        • container lifecycle management (e.g. starting/killing)
          • authenticate container leases
          • configure the environment for the contianer (e.g. init its monitoring subsystem with the resource constraints specified in the lease)
          • manage containers' dependencies (e.g. copy all the necessary dependencies to local storage)
            • dependencies may be shared b/w containers in an application, or bw/w containers launched by the same tenant, or even b/w tenants (specified in the CLC)
            • eventually garbage collects dependencies not in use by running containers
          • kill containers as directed by the RM or the AM
            • e.g. may be killed when RM reports its owning app as completed, when the scheduler decideds to evit it for another tenant, when the NM detects that the container exceeded the lease limits, etc.
            • e.g. AMs may request containers to be killed when the corresponding work isn't needed any more
            • whenever a container exits, the NM will clean up its working directory in local storage
            • e.g. when an app completes, all resources owned by its containers are discarded on all nodes , including any of its processes still running in the cluster
          • monitor containers' execution
          • provide services to containers
            • e.g. a log aggregation service t hat will upload data written by the application to stdout and stderr to HDFS once the app completes
            • admin may configure the NM with a set of pluggable, auxiliary services
              • a process may produce data that persist beyond the lifecycle of the container, to be managed by the node
              • e.g. in Hadoop MapReduce, intermediate data are transferred b/w map and reduce tasks using an auxiliary service
              • CLC allows AMs to address a payload to a auxiliary service. MapReduce apps use this channel to pass tokens that authenticate reduce tasks to the shuffle service
        • monitor resource availability. NM periodically monitors the health of the physical nodes and runs an admin configured script frequently that in turn can point to any HW/SW issues
          • e.g. memory, CPU, etc. allocated for YARN
          • e.g. local disks
          • when issue detected, NM changes its state to be unhealthy and reports RM about the same which then makes a scheduler specific decision of killing the containers and/or stopping future allocation on this node till the health issue resolved
          • send status in the heartbeat msg to the RM and receives instructions
        • report faults
      • All containers (NMs, AMs) are described by a container launch context (CLC)
        • a map of env vars, dependencies stored in remotely accessible storage, security tokens, payloads for NM services, the command necessary to create the process
        • can also include crednetials to authenticate the download if required
    • Flow
      • Jobs are submitted to the RM via a public submission protocol and go through an admission control phase during which security credentials are validated and various operational and administrative checks are performed
      • Accepted jobs are passed to the scheduler to be run
      • Once the scheduler has enough resources,
        • the application state transitioned from "accepted" to "running";
        • the RM allocates a container for the AM and spawn it on a node in the cluster
        • a record of accepted applications is written to persistent storage and recovered in case of RM restarts or failure
      • To obtain containers, AM issues resource requests to the RM
        • the request includes specification of locality preferences, properties of the containers
      • the RM will attempt to satisfy the resource requests coming from each application based on resource availability and shceduling policies
      • When a resource is allocated on behalf of an AM, the RM generates a lease for the resource, which is pulled by a subsequent AM heartbeat
      • YARN use a token-based security mechanism to guarantee the lease authenticity when the AM presents the container lease to the NM
      • Once the AM discovers that a container is available for its use, it encodes an app-specific launch request with the lease
      • if needed, running containers may communicate directly with the AM through an application-specific protocol to report status, liveness, and receive framework-specific commands
        • YARN neither facilitates nor enforces this communication
    • Overall, YARN deployment provides a basic, yet robust infra for lifecycle management and monitoring of containers, while app-specific semantics are managed by each framework
    • YARN only secures its deployment and it is the application's responsibility to secure its operations if the app wants to expose a service or wires a communication graph
    • Fault tolerance and availability
      • RM remains a single point of failure in YARN's architecture
        • the RM recovers from its own failures by restoring its state from a persistent store on initialization
        • once the recovery done, it kills all the containers running in the cluster (including AMs, NMs)
        • it then launches new instances of each AM
          • if the framework supports recovery, the platform will automaticall yrestore users' pipelines
      • NM failure
        • when a NM fails, the RM detects it by timing out its heartbeat response, marks all the containers running on that node as killed, and reports the failure to all running AMs
        • if the fault is transient, the NM will resync with the RM, cleanup its local state, and continue
        • in both cases, AMs are responsible for reacting to node failures, potentially redoing work done by any containers running on that node during the fault
      • AM failure
        • Since AMs run in the cluster, its failure does not affect the availability of the cluster, but application hiccup due to AM failure is more probable than in Hadoop 1.x.
        • RM may restart the AM if it fails, though the platform offers no support to
          • restore the AMs state
          • sync with its own running containers...
        • e.g. Hadoop MapReduce AM will recover its completed tasks, but running tasks, or tasks that completed during AM recovery, will be killed and re-run
      • Container failure
        • completely left to the frameworks...
        • the RM collects all container exit events from the NMs and forward them to the corresponding AMs in a heartbeat response
        • e.g. the Hadoop MapReduce AM already listens to these notifications nad retries map or reduce tasks by requesting new containers from the RM
