Borg, Omega, and Kubernetes
  • Lessons learned from 3 container management systems over a decade
    • 3 container management systems that are built by Google and each system was heavily influenced by its predecessors
      • Borg
        • motivation: built to manage both long-running services and batch jobs (replace the Babysitter and Global Work Queue systems)
        • Borg shares machines b/w these two types of applications as a way of increasing resource utilization and thereby reducing costs
          • sharing is possible because container support in the linux kernel was becoming available and therefore enabled better isolation b/w latency-sensitive user-facing services and CPU-hungry batch processes
        • Driven by the needs of different teams inside Google, the app team and infra team developed a broad ecosystem of tools and services for Borg
          • provide mechanisms for configuring and updating jobs
          • predict resource requirements
          • dynamically push configuration files to running jobs
          • service discovery and load balancing
          • auto-scaling
          • machine lifecycle management
          • quota management
          • ...
          • => a somewhat heterogeneous, ad-hoc collection of systems that Borg users had to configure and interact with, using several different configuration languages and processes
        • Borg remains the primary container-management system within Google because of its scale, breath of features, and extreme robustness
      • Omega
        • motivation: we want to improve the software engineering of the Borg ecosystem
        • it applied many of the patterns that had proved successful in Borg but was built from the ground up to have a more consistent, principled architecture
        • it stores the cluster states in a centralized Paxos-based transaction-oriented store
          • the store will be accessed by the different parts of the cluster control plane
          • the store uses optimistic concurrency control to handle the occasional conflicts
        • the decoupling allowed the Borgmaster's functionality to be broken into separate components that acted as peers, rather than funneling every change through a monolithic centralized master
      • Kubernetes
        • motivation: external developers were becoming interested in Linux containers and Google had developed a growing business selling public-cloud infrastructure. Kubernetes is open source. The goal is to make it easy to deploy and manage complex distributed systems while still benefiting from the improved utilization that containers enable
        • its core is a shared persistent store with components watching for changes to relevant objects
        • state in Kubernetes is accessed exclusively through a domain-specific REST API that applies higher-level versioning, validation, semantics, and policy, in support of a more diverse array of clients
        • it was developed with a stronger focus on the experience of developers writing applications that run in a cluster
    • Introduction to Containers
      • the first containers just provide isolation of the root file system via `chroot`
      • then FreeBSD jails extend this to additional namespaces such as process IDs
      • Solaris subsequently pioneered and explored many enhancements
      • Linux control gorups (cgroups) adopted many of these ideas and development in this area continues today
      • containers provide
        • Resource isolation and resource management
          • enable users to improve resource utilization
          • e.g. Borg uses containers to co-locate batch jobs with latency-sensitive, user-facing jobs on the same physical machine. The user-facing jobs reserve more resources than they usually need, allowing them to handle load spikes and fail over. The mostly-unused resources can be reclaimed to run batch jobs colocated in the same physical machine
          • Robust kernel-level resource isolation prevents processes from interfering with one another
          • the isolation is not perfect: containers cannot prevent interference in resources that the OS does not manage e.g. level 3 processor caches, memory bandwidth
            • => containers need to be supported by an additional security layer such as Virtual Machines to protect against the kinds of malicious actors found in the cloud
      • container images
        • include the files that make up the application that runs inside the container
        • Google uses MPM (Midas Package Manager) to build and deploy container images
      • the relationships b/w the isolation mechanism and MPM packages
        • "symbiotic" relationship
        • similar to the relationship b/w Docker daemon and the Docker image registry
    • Application-oriented infrastructure
      • Containerization transforms the data center from being machine-oriented to being application-oriented
      • Eg 1: containers encapsulate the application environment, abstracting away many details of machines and OSs from the application developer and the deployment infrastructure
      • Eg 2: well-designed containers and container images are scoped to a single application, and therefore managing containers means managing applications rather than machines. This shift of management APIs dramatically improves application deployment and introspection
      • what's the purpose of "cgroup", "chroot", "namespace" facilities in the kernel?
        • they are used to protect the application from noisy, nosey, and messy neighbors
      • Combining these with container images creates an abstraction that application is isolated from the heterogeneous OSs on which they run
      • the decoupling of image and OS makes it possible to provide the same deployment environment in both development and production, which in turn improves deployment reliability and speeds up development by reducing inconsistencies and friction
        • a hermetic container image that can encapsulate almost all of an application's dependencies into a package that can be deployed into the container
          • the only external dependencies will be on the linux kernel system-call interface
          • apps still be exposed to churn in the OS interface, particularly in the wide surface area exposed by socket options, /proc, and arguments to ioctl calls
        • To achieve such "hermetic" images...
          • In Borg, program binaries are statically linked at build time to know-good lib versions hosted in the company-wide rerpo.
            • Even so, the Borg container image is not quite airtight than it could have been, because apps share a so-called base image that is installed once on the machine rather than being packaged in each container
            • the base image contains utilities such as tar and the libc library, so upgrades to the base image can affect running applications and may lead to significant source of trouble (hard to debug)
          • Modern container image formats such as Docker and ACI harden this abstraction further and get closer to the hermetic ideal by eliminating implicit host OS dependencies and require an explicit user command to share image data b/w containers
      • load balancers don't balance traffic across machines, but balance across application instances
      • logs are keyed by application, not machine, so they can easily be collected and aggregated across instances without pollution from multiple apps or system operations
      • application failure detection and failure cause attribution can be easily achieved without having to disentangle them from machine-level signals
    • Containers as the unit of management
      • Benefits
        • relieves application developers and operation teams from worrying about specific details of machines and OSs
        • provides the infra team flexibility to roll out new hardware and upgrade OSs with minimal impact on running apps and their developers
        • ties telemetry collected by the management systems to applications rather than machines, which dramatically improves application monitoring and introspection, esp when scale-up, machien failures, or maintenance cause apps instances to move
        • provides convenient points to register generic APIs that enable the flow of information b/w the management system and an application without either knowing much about the particulars of the other's implementation
          • In Borg, this API is a series of HTTP endpoints attached to each container
        • the container-management system can communicate information into the container such as resource limits, container metadata for propagation to logging and monitoring, and notices that provide graceful-termination warnings in advance of node maintenance
        • provide application-oriented monitoring in other ways
          • e.g. through linux kernel cgroups (which provides resource-utilization data about the application), which can be extended with custom metrics exported using HTTP APIs
            • such data enable the dev of generic tools like an auto-scaler, cAdvisor, that can record and use metrics without understanding the specifics of each application
          • enable finer-grained reporting and control of metrics and logs
      • in reality, we use nested containers that are co-scheduled on the same machine. The outermost one provides a pool of resources while the inner ones provide deployment isolation
        • In Borg, the outermost container is called a resource allocation, or alloc
          • it allows top-level application container to run outside allocs, which has been a source of much inconvenience
        • In Kubernetes, the outermost container is called a pod
          • it always run an application container inside a top-level pod even if the pod contains only a single container
          • a common use pattern is, for a pod to hold an instance of a complex application. the major part of the app sits in one of the child containers, and other child containers run supporting fucntions such as log rotation or click-log offloading to a distributed file system
            • such separation of different components makes it easy to have different teams develop the distinct pieces of functionality and it improves the robustness, composability, and fine-grained resource isolation
              • robustness: the offloading continues even if the main application gets wedged
              • composability: easy to add a new small support service because it operates in the private execution env provided by its own container
              • fine-grained resource isolation: each runs in its own resources so the logging system can't starve the main app, or vice versa
    • Orchestration is the beginning, not the end
      • container management per se was just the beginning of an env for developing and managing reliable distributed systems
        • e.g. many different systems have been built in, on, and around Borg to improve upon the basic container-management services that Borg provided
          • e.g. naming and service discovery, master election, application-aware load balancing, horizontal and vertical autoscaling, rollout tools that manage the careful deployment of new binaries and configuration data ......
      • Kubernetes attempts to avert the increased complexity by adopting a consistent approach to its APIs
        • e.g. every Kubernetes object has 3 basic fields in its description: ObjectMetadata, Specification and State
          • ObjectMetadata is the same for all objects in the system, it contains info such as the object's name, UID, version number, labels..
          • Spec and Status vary by object type, but their concepts does not
            • Spec is used to describe the desired state of the object
            • Status provides read-only info about the current state of the object
        • such uniform API provides many benefits
          • learning the system is simpler since similar info applies to all objects
          • writing generic tools that work across all objects is simpler which in turn enables the dev of a consistent user experience
          • learning from Borg and Omega, Kubernetes is built from a set of composable building blocks that can readily be extended by its users and a common API and object-metadata structure makes that much easier
            • e.g. the pod API is usable by people, internal Kubernetes components, and external automation tools
          • "consistency"
            • Kubernetes is being extended to enable users to add their own APIs dynamically alongside the core Kubernetes functionality
            • consistency also achieved by decoupling in the Kubernetes API
              • "separation of concern" b/w API components means higher-level services all share the same common basic building blocks
                • e.g. the separation b/w the Kubernetes replication controller and its horizontal auto-scaling system
              • decoupling ensures that multiple related but different components share a similar look and feel
                • e.g. Kubernetes has 3 different forms of replicated pods
                  • ReplicationController that run-forever replicated containers e.g. web servers
                  • DaemonSet that ensures a single instance on each node in the cluster e.g. logging agents
                  • Job that is a run-to-completion controller that knows how to run a possibly parallelized batch job from start to finish
                  • all 3 controllers rely on the common pod object to specify the containers they wish to run
            • consistency also achieved by common design patterns for different Kubernetes components
              • the idea of a reconciliation controller loop is shared throughout Borg, Omega, and K8s to improve the resiliency of a system
                • it compares a desired state against the observed state and takes actions to converge the observed and desired states
                • all actions are based on observation rather than a state diagram, it is much more robust to failures and perturbations. when a controller fails or restarts, it simply picks up where it left off
        • the design of k8s as a "combination of microservices and small control loops" is an example of control through "choreography"
          • achieving the desired emergent behavior by combining the effects of separate autonomous entities that collaborate
          • this is a conscious design choice in contrast to a centralized orchestration system, which may be easier to construct at first but tends to brittle and rigid over time, esp in the presence of unanticipated errors or state changes
