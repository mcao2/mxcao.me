VMware Distributed Resource Management: design, implementation, and lessons learned
  • Automated resource management
    • is critical for reducing the operational costs of virtualized environments
    • requires
      • performance isolation among VMs
      • handle resource fragmentation across physical hosts and optimize scheduling for multiple resources
      • utilize the underlying hardware infra efficiently
    • VMware's solutions
      • DRS (Distributed Resource Scheduler)
        • manages the allocation of physical resources to a set of VMs deployed in a cluster of hosts, each running the VMware ESX hypervisor
        • maps VMs to hosts and performs intelligent load balancing in order to improve performance and to enforce both user-specified policies and system-level constraints
        • supports a "what if" mode that make it possible to evaluate the impact of changes in workloads or cluster configuration
        • significantly improves the overall performance of VMs running in a cluster
      • DPM (Distributed Power Management)
        • extends DRS with the ability to reduce power consumption by consolidating VMs onto fewer hosts
        • recommends evacuating and powering off hosts when CPU and memory resources are lightly utilized
        • recommends power on hosts appropriately as demand increases or as required to satisfy resource-management policies and constraints
        • in clusters with non-trivial periods of lowered demand, DPM significantly reduces server power consumption
  • Introduction/Background
    • Initially, rapid adoption of virtualization was fueled by significant cost saving resulting from server consolidation.
    • later, more advanced VM capabilities such as cloning, template-based deployment, checkpointing, live migration of running VMs led to more agile IT infra
    • it now became much easier to create and manage VMs
    • such ease of deployment leads to increasingly large VM installations
    • also, HW tech continue to improve, causing consolidation ratio to rise
    • but...
      • the operational costs of managing VMs now take a significant fraction of overall costs for data centers using virtualization
        • ideally, the complexity of managing a virtualized env should also benfit from consolidation, scaling with the number of hosts rather than the number of VMs. otherwise it would be hard due to sharing and contention
        • in practice, large fraction of operational costs were related to
          • inherent complexity of determining good VM-to-host mappings
          • deciding when to use vMotion (VMware's live migration tech) to rebalance load by changing those mappings
          • the difficulty of this problem is exacerbated by the fragmentation of resources across many physical hosts and the need to balance the utilization of multiple resources simultaneously
          • also, administrators need a reliable way to specify resource-management policies
            • need expressive resource controls to prioritize VMs of varying importance in order to isolate and control the performance of diverse workloads competing for the same physical hardware
    • DRS is designed as a way to help reduce the operational complexity of running a virtualized datacenter
      • enable managing a cluster containing many potentially-heterogeneous hosts as if it were a single pool of resources
      • DRS provides:
        • a cluster abstraction for managing a collection of hosts as a single aggregate entity with combined processing and memory resources
        • a powerful resource pool abstraction which supports hierarchical resource management among both VMs and groups of VMs. DRS provides rich set of controls to express policies that manage resource contention for each level in the hierarchy
        • automatic initial placement, i.e. assign a VM to a specific host within the cluster when it is powered on
        • dynamic load balancing of both CPU and memory resources across hosts in response to dynamic fluctuations in VM demands, and changes to the physical infra
        • custom rules to constrain VM placement e.g. affinity and anti-affinity rules among VMs and b/w VMs and hosts
      • Experimental results and production adoption showed that
        • DRS is able to meet resource management goals while improving the utilization of underlying hardware resources
        • DPM can save significant power in large configurations with many hosts and VMs
  • DRS Resource Model & associated resource controls
    • def: resource model explains the capabilities and goals of a resource-management solution
    • DRS's flexible set of resource controls enable a wide range of resource-management policies to be specified to provide differentiated QoS to groups of VMs
    • Basics
      • allow users to express allocation in terms of either absolute VM allocation or relative VM importance
      • control knobs are provided at the individual host level by the ESX hypervisor
      • DRS extend such controls to the cluster setting (which consists of multiple ESX hosts), allowing them to be managed as a single entity
      • basic resource controls
        • Reservation
          • a reservation specifies a minimum guaranteed amount of a certain resource (i.e. lower bound that applies even when this resource is over-committed heavily)
          • expressed in absolute units e.g. MHz for CPU, MB for memory
          • admission control during VMpower-on ensures the sum of reservations for a resource does not exceed its total capacity
        • Limit
          • a limit specifies the upper bound on the consumption of a certain resource even when this resource is under-committed
          • expressed in concrete absolute units e.g. MB for memory
          • a VM is prevented from consuming more than its limit even if that leaves some resources idle
        • Shares
          • shares specify relative importance
          • expressed using abstract numeric values
          • a VM is entitled to consume resources proportional to its share allocation and it is guaranteed a minimum resource fraction equal to its fraction of the total shares when there's contention for a resource
      • Reservation and Limits controls avoid VMs from unacceptable or unpredictable performance
      • DRS focuses primarily on CPU and memory resources, but similar controls for IO resources, network and storage bandwidth also have been validated
    • Resource pools
      • a logical resource pool facilitates flexible resource-management policies for groups of VMs
      • def: a logical resource pool is a container that specifies an aggregate resource allocation for a set of VMs and it is a named object with associated settings for each managed resources
      • admission control is performed at the pool level
        • the sum of the Reservations for a pool's children must not exceed the pool's own reservation
      • it can be configured in a flexible hierarchical organization where each pool has an enclosing parent pool and children that may be VMs or sub-pools
      • resource pools are useful for dividing or sharing aggregate capacity among groups of users or VMs
        • e.g. use resource-pool hierarchies to mirror human organizational structures
        • admin can allocate resources in bulk to sub-admins using sub-pools
        • separate per-pool allocation provide both isolation b/w pools and sharing within pools
          • in a resource pool hierarchy, shares are meaningful only with respect to siblings
          • each pool effectively defines a scope within which share values are interpreted
    • Resource pool divvy
      • def: the process of computing the entitled reservation, limit, and shares of its child sub-pools and VMs as divvying
        • the computation incorporates user-specified resource control settings as well as per-VM and aggregate workload demands
      • divvying starts from the root of the hierarchy and ultimately ends with the VMs at its leaves
      • DRS uses the resulting entitlements to update host-level settings properly, ensuring that controls such as reservations and limits are enforced strictly by the ESX hypervisor
      • DRS also uses host-level entitlement imbalances to inform VM migration decisions
      • the parent pool values need to be divided among the children based on the user-set values of reservation, limit, shares and the actual runtime demands of the children
      • divvy operations
        • reservation-divvy
        • limit-divvy
        • share-divvy
      • DRS carries out these divvy operations periodically, reflecting current VM demands
      • DRS also initiates divvy operations in response to changes in resource allocation settings
      • two phases
        • bottom-up phase: starts with the demand values of the individual VMs (leaf nodes) and then accumulates aggregate demands up the resource pool tree
        • top-down phase: reservation and limit values at each parent are used to compute the respective resource settings for its children such that the constraints are met
          • 1. child allocation are in proportion to their shares
          • 2. each child is allcoated at least its own reservation
          • 3. no child is allocated more than its own limit
    • DRS design
      • 4 key resource-management operations
        • computes the amount of resources to which each VM is entitled based on the reservation, limit and shares values and the runtime demands for all related VMs and resource pools
        • recommends and performs migration of powered-on VMs to balance load across hosts in a dynamic env where VM's resource demands may change over time
        • optionally saves power by invoking DPM
        • performs initial placement of VMs onto hosts so that a user does not have to make manual placement decisions
      • DRS load balancing - the core to supports the DRS resource model
        • is invoked periodically to satisfy cluster constraints and ensure delivery of entitled resources
        • can be invoked by users when they make cluster configuration changes
      • DRS initial placement - build on the DRS load-balancing model
        • assign VMs to a host within a cluster
        • shares code with DRS load balancing to ensure that the placement recommendations respect constraints and resource entitlements
      • Constraints
        • fundamental constraint: VM-to-Host compatibility (i.e. the ability of a host to satisfy a VM's execution requirements)
          • DRS respect this constraint during load balancing and VM placement
        • DRS supports enforcing a set of constraints to handle different use cases
          • e.g. co-located of VMs for performance,
          • placement of VMs on separate HW for availability and fault isolation,
          • affinity to particular HW to handle licensing issues
        • DRS also handles
          • evacuation of hosts the user has requested to enter maintenance or standby mode,
          • preservation of sparse resources for failover, and
          • the role of special VMs that provide services to other VMs
        • DRS respects constraints during VM initial placement and runs a pass prior to load balancing to correct violations of DRS-enforced constraints
          • reporting errors for any of those constraints that cannot be corrected
        • Constraint 1: Affinity Rules
          • DRS supports VM-to-VM or VM-to-Host rules which are used for a variety of common business scenarios.
          • VM-to-VM anti-affinity rules define a set of VMs that are to be kept on separate hosts
            • e.g. used for availability and are mandatory i.e. DRS will not make any recommendation that violates them
          • VM-to-VM affinity rules define a set of VMs that are to kept on the same host
            • e.g. used to enhance the performance of communicating VMs because intra-host VM-to-VM networking is optimized to perform in-memory packet transfers without using NIC hardware
            • these rules are mandatory for load balancing, but DRS will violate them if necessary to place VMs during power -on
          • VM-to-Host rules define a set of VMs being affine or anti-affine with a set of hosts
            • can be specified as mandatory or preferred
              • preferred means they are enforced unless they engender additional constraints violations or cannot be respected without causing host overutilization
              • useful for a variety of reasons
              • e.g. Mandatory VM-to-Host affinity rules are often used to enforce licensing, to associate VMs requiring a host-based software license with the hosts having that license
                • are represented in the vm-to-host compability info passed to DRS and hence are handled as a fundamental constraint
              • e.g. Preferred VM-to-Host affinity or anti-affinity rules are often used to manage availability and/or site locality
                • are represented as alternative vm-to-host compability info
                • DRS handles such rules by running a "what-if" pass with the preferred rules treated as mandatory
                  • accepted only if the resulting cluster state has no constraint violations or overutilized hosts
        • Constraint 2: High availability
          • DRS supports HA in two ways
            • by preserving powered-on idle resources to be used for VM restart in accordance with the HA policy
            • by defragmenting resources in the cluster when HA restart cannot find sufficient resources
